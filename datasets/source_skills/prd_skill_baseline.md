# PRD Writer

帮助产品经理和团队撰写结构清晰、内容完整的产品需求文档。

## PRD 核心问题

在开始撰写前，PRD 必须能回答以下问题：
- 我们在解决什么问题？
- 我们在为谁解决问题？
- 这个问题为什么重要？
- 成功的标准是什么？
- 具体需要构建什么？

## 章节结构（按顺序输出）

### 1. 背景与现状
说明为什么需要做这个产品或功能，包括：
- 当前现状与已识别的问题或低效点
- 业务动机
- 数据或调研支持（如有）

### 2. 目标
定义产品要达成的目标，目标应具体、可衡量，并与业务战略一致。例如：
- 在三个月内提升用户参与度 20%
- 降低任务完成时间
- 提高转化率

### 3. 目标用户与用户画像
识别产品的目标用户，描述其特征、需求和行为，包括：
- 用户分群
- 用户画像（Persona）
- 核心使用场景

### 4. 用户场景与用例
描述用户在真实场景中如何使用产品。例如：
- 销售经理希望在 CRM 系统中跟踪客户线索
- 用户打开应用并收到个性化推荐

用"作为[用户类型]，我希望[功能]，以便[价值]"的格式描述用户故事。

### 5. 功能需求
描述系统需要实现的具体功能，包括：
- 功能描述
- 输入与输出
- 用户交互方式
- 业务逻辑
- 边界情况

格式示例：
- 系统应允许用户通过邮箱和密码登录
- 登录失败时系统应显示错误提示

### 6. 非功能需求
描述系统的性能表现，包括：
- 性能（如响应时间）
- 可扩展性
- 稳定性
- 安全性
- 可用性

### 7. 用户体验与设计说明
描述界面交互要求，确保：
- 产品易于使用
- 使用流程一致
- 设计符合用户需求

可包含页面跳转逻辑、关键交互说明或低保真描述。

### 8. 假设与约束
说明规划过程中所做的假设以及可能影响实施的限制条件。例如：
- 假设用户具备稳定的网络连接
- 工程资源有限

### 9. 依赖关系
说明对其他系统、团队或外部服务的依赖。例如：
- 需要接入支付系统
- 依赖第三方 API 可用性

### 10. 验收标准
每个功能点对应一条可验证的完成标准，格式为：
"系统应[行为]"

### 11. 度量指标与成功标准
定义产品上线后如何衡量成功，常见指标包括：
- 用户采用率
- 留存率
- 转化率
- 错误率

### 12. 发布计划与时间线
描述开发与发布的时间安排，包括：
- 里程碑
- 阶段划分
- 截止时间

### 13. 非目标
明确列出本次不做的内容，避免范围蔓延。

### 14. 开放问题
列出尚未解决的问题或待确认事项。

## 输出要求

- 使用 Markdown 格式，章节标题使用 ## 或 ###
- 语言与用户输入保持一致
- 每个章节必须有实质内容，不得只写标题
- 章节顺序严格按照上述结构，不得调整
