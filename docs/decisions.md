# 技术决策日志

本文档记录项目七个关键决策时刻的思考过程。
每个决策遵循「问题 → 思考 → 决策 → 发现」的结构。

---

## D1：为什么用 Skill Gain 而不是绝对分数

**问题**

如何衡量一个 Skill 的真正价值？直接让 LLM 给 PRD 打分可以吗？

**思考**

绝对分数有一个根本缺陷：它无法区分"Skill 本身好"和"模型基础能力强"。

在 B2B 场景下，Kimi K2.5 内置了大量企业产品知识，不用任何 Skill 就能生成 0.689 分的 PRD。如果加了 Skill 变成 0.699，绝对分数看起来有提升，但这个 Skill 真的带来了增量价值吗？

更重要的问题是可交付性：一份 0.65 分的 PRD 能交付给开发团队吗？如果不能，那么从 0.3 提升到 0.65 的 Skill，和从 0.65 提升到 0.7 的 Skill，在业务价值上是完全不同的。

**决策**

设计三层评估体系：

- Skill Gain（归一化增量）：衡量相对提升，消除 baseline 水平差异
- Quality Floor（质量底线）：确保绝对质量达到可交付标准
- Efficacy Matrix（效用矩阵）：两个维度合并为可操作的 HIGH/MEDIUM/LOW 标签

**发现**

B2B 场景验证了这个设计的必要性。zero_shot Skill 的绝对分数 0.699 看起来不错，但 Gain 只有 +0.032，说明 Skill 几乎没有带来增量价值。如果只看绝对分数，会得出错误结论。

## D2：为什么用不同模型分别承担生成和评估

**问题**

用谁来评估 Kimi 生成的 PRD 质量？

**思考**

最直接的方案是用 Kimi 评估自己生成的 PRD，但这存在自我评估偏差的风险：模型可能对自己的输出风格更熟悉，倾向于给出更高的分数，或者在边界案例上产生系统性偏差。

另一个问题是批量评估效率。如果每条断言单独调用一次 Claude，B1 有 5 条 LLM 断言，每次 run 就要调用 5 次，7 个 case × 3 runs × 2 conditions = 210 次 Claude 调用，成本和时间都难以接受。

**决策**

- 被评估者：Kimi K2.5（Moonshot）负责生成 PRD
- 评估者：Claude Sonnet 4.6（Anthropic）负责 Judge

两个不同公司、不同架构的模型，消除自我评估偏差。同时把同一份 PRD 的所有 LLM 断言合并为一次 Claude 调用，返回 JSON 数组，将调用次数从 210 次降到 42 次。

**发现**

实验中验证了 Claude Judge 比 Rule 断言更接近真实情况。对于"验收标准是否存在"这个断言，Claude 给出 confidence=0.75（判断存在），而 Rule 断言因为关键词不匹配判定失败。人工复核确认 Claude 是对的。

## D3：为什么从多进程改为异步 IO

**问题**

如何实现高效的并发评估，让多个 run 同时进行？

**思考**

最初设计是 `ProcessPoolExecutor` 多进程并发，每个进程独立调用 API。但在 Windows spawn 模式下，`tenacity` 的 `@retry` 装饰器内部使用了 `threading.RLock`，无法被 pickle 序列化传递给子进程，导致进程池在启动时就崩溃。

评估任务的本质是 IO 密集型——大部分时间在等待 Kimi API 响应，CPU 几乎空闲。多进程适合 CPU 密集型任务，对 IO 密集型任务来说是"杀鸡用牛刀"，而且引入了 Windows 平台的兼容性问题。

**决策**

- 放弃多进程，改用 `asyncio + aiohttp` 异步方案
- 直接调用 Moonshot HTTP 接口，不依赖 openai SDK，减少一层封装
- `asyncio.Semaphore(3)` 控制最大并发数，既提升速度又避免 429 限流

**发现**

异步方案稳定运行，解决了 Windows 兼容性问题。Semaphore=3 的并发控制在速度和稳定性之间取得了良好平衡，整个 B1 实验（7 cases × 3 runs × 2 conditions）耗时约 29 分钟。

## D4：为什么接受 temperature=1.0 的代价

**问题**

Kimi K2.5 流式传输强制要求 temperature=1.0，而原设计是 temperature=0.6 的稳定模式。

**思考**

temperature=1.0 会增加输出随机性，导致同一个 prompt 的多次输出差异较大，评估一致性下降。这直接体现在实验数据中——部分 case 的 Cohen's Kappa=0.000，说明 3 次 run 的结果分布不稳定。

但不用流式传输的代价更大：长 PRD 生成需要 2-3 分钟，非流式模式要等全部内容生成完才返回响应。在韩国网络环境下访问中国区 Kimi 服务器，加上代理延迟，60 秒的超时设置几乎必然触发，导致大量请求失败。

这是一个主动的工程权衡，不是被动接受：**"实验可运行"比"结果完美稳定"更重要，但必须在报告中明确标注这一局限性。**

**决策**

- 接受 temperature=1.0，使用流式传输解决超时问题
- 用 n_runs=3 多次采样取平均，缓解单次随机性
- 在报告中明确标注 Kappa=0.000 的情况及其原因
- 不将 consistency 和 Kappa 作为主要评估指标，仅作参考

**发现**

n_runs=3 基本能反映真实水平，Gain 指标在多次实验中保持相对稳定。但 Kappa=0.000 的情况在搜索功能、电商购物车等 case 仍然出现，说明这些场景的评估稳定性有待提升，需要更多采样次数。

## D5：为什么主动打开样本文件做人工复核

**问题**

自动评估的分数可信吗？框架本身有没有系统性偏差？

**思考**

整个评估流程是一个黑箱——只能看到最终的数字，不知道 Kimi 实际生成了什么内容，不知道 Claude 为什么给出这个判断，更不知道 Rule 断言的通过/失败是否符合实际。

如果评估本身有问题，所有基于这些数字的结论都是错的。一个看起来"差异不显著"的结果，到底是 Skill 真的没有价值，还是评估框架低估了它？

**决策**

主动要求加入样本保存机制，将每次 run 的完整 PRD 内容 + 每条断言的判断结果和 confidence 值保存为 JSON 文件，存入 `experiments/samples/`。然后手动打开样本，逐条对照断言判断和实际 PRD 内容。

这个决策的核心是：**把评估框架本身也纳入质疑范围，而不是假设框架是正确的。**

**发现**

人工复核直接揭示了 Rule 断言的系统性低估。一份包含完整"页面布局与交互"章节的 PRD 被 b1_07 断言判定失败，因为关键词列表里没有这个中文标题。去掉误判后，该 PRD 的实际质量约 0.762，而自动评估只给了 0.578，偏差约 0.18。

这个发现改变了整个实验的解读方向：能力提升不显著，部分原因是 Skill 确实效果有限，但部分原因是断言本身在系统性低估中文 PRD 的质量。

## D6：为什么保留 v1 断言而不是直接修复

**问题**

发现 v1 断言有系统性偏差后，为什么不直接修改关键词，而是新建 v2 版本？

**思考**

直接修改 v1 会破坏数据的可比性。所有已经用 v1 跑完的实验数据，无法和修改后的 v1 数据进行比较，历史数据的价值会丧失。

更重要的是，v1 和 v2 测量的是不同维度，两者都有存在价值：

- v1：评估 Skill 是否引导输出英文标准格式的 PRD（格式合规性）
- v2：评估 Skill 是否引导输出实质内容完整的 PRD（内容存在性）

如果只保留 v2，就失去了"Skill 对格式规范化的影响"这个维度的数据。

**决策**

新建 v2 断言文件，两个版本并存，通过 `--assertions-version v1/v2` 参数切换。v1 历史数据完整保留，v2 作为修正后的参照版本。让评估偏差本身成为可观测、可量化的数据。

**发现**

v2 实验揭示了一个重要现象：v2 下 baseline 得分提升幅度（+0.145）远大于 with_skill 的提升幅度（+0.056）。这说明 v1 对 baseline 的低估更严重——Kimi 在没有 Skill 时也能用中文惯用标题写出完整内容，而 Skill 引导的输出反而更接近英文标准格式，被 v1 低估程度相对较轻。

## D7：为什么引入 prd-taskmaster 作为 baseline 参考

**问题**

baseline Skill 应该写得多好？用什么标准衡量 baseline 的质量？

**思考**

如果 baseline Skill 太简陋，zero_shot 和 few_shot 都能轻松超越它，实验缺乏区分度。如果 baseline 太复杂，被评估的 Skill 可能都达不到 baseline 水平，Gain 全部为负。

更根本的问题是：baseline 应该代表什么？

我认为 baseline 应该代表"当前已知最好的人类设计"，这样实验回答的才是真正有价值的问题——Agent 能否生成接近甚至超越最优人类 Skill 的 Skill？这比"Agent 能否超越一个随意写的简陋 Skill"更有意义。

**决策**

主动搜索市面上最好的开源 PRD Skill，找到 prd-taskmaster。用 Kimi K2.5 思考模式提取其核心设计思想，包括质量分级（EXCELLENT/GOOD/ACCEPTABLE/NEEDS WORK）、13 项验证检查思路、用户测试检查点概念，融入 baseline 设计。

同时保留 baseline 的"刻意不完整"特性——不加入 B2B/C端/内部工具差异化指引，不加入多步骤任务拆解，为 few_shot Skill 留出改进空间，确保实验有区分度。

**发现**

few_shot Skill 参考了这个 baseline 和 prd-taskmaster 的设计思想，在 B1 结构合规性上取得了比 zero_shot 更高的 Gain（+0.326 vs +0.231）。在 B2 的 B2B 场景中，few_shot 的专门差异化指引将 Gain 从 +0.032 提升到 +0.270，直接验证了高质量参考设计的价值。
